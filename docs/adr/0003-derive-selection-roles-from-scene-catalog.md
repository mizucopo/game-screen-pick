# Derive Selection Roles From Scene Catalog

Scene selection needs to reduce cinematic scenes while allowing more variants for recurring gameplay patterns, and those categories vary by game. We decided to make this the default selection behavior and let Ollama assign a scene selection role (`ordinary`, `cinematic`, or `recurring_gameplay`) to each scene catalog entry instead of using a fixed slug list, manual quota table, user-selected mode, or per-image role classification, because the existing catalog flow already adapts to each input set and selection intent. Images inherit the role of their classified `scene_slug`, and `other` is normalized to `ordinary` even if Ollama returns a different role. Catalog representatives should keep the current 24-image limit but include high-quality frames, diverse frames, and frequent visual patterns so recurring gameplay can be recognized during catalog creation without increasing Ollama cost. Cinematic scenes should share a soft cap of 10% of the requested output, with a minimum of one image and overflow allowed when non-cinematic candidates cannot fill the request. The cinematic cap should be applied during scene target allocation, with freed slots redistributed to ordinary and recurring gameplay scenes before final similarity selection. Recurring gameplay should expand variants by relaxing similarity earlier than ordinary scenes while still avoiding near-identical frame flooding. Selection roles should also be surfaced in reports so users can explain why cinematic scenes were limited or recurring gameplay variants were expanded, and included in Ollama classification cache keys because role changes affect final selection behavior.
